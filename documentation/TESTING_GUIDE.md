# PathWise — Testing Guide

This guide covers how to verify PathWise's adaptive logic, UI flows, and offline behaviour. The project uses TypeScript compilation checks as its primary build-time validation, with manual and script-based testing for runtime correctness.

---

## 1. Build Validation (Primary Check)

The fastest way to catch regressions is the TypeScript compiler:

```bash
npm run build
```

A successful build (`tsc -b && vite build`) with **zero TypeScript errors** is the baseline requirement before any commit. This catches:
- Type mismatches in BKT, SM-2, and mastery computations
- Missing props or incorrect interface shapes
- Import errors and unused variables (`verbatimModuleSyntax` enforced)

## 3. BKT Algorithm Verification

Manually verify the BKT update logic in `src/lib/bkt.ts`:

**Test case — correct answer, low prior:**
```typescript
import { updateBKT, createDefaultBKT } from './src/lib/bkt';

const initial = createDefaultBKT('user1', 'chapter1', 'skill1');
// pKnow should start at ~0.3 (pLearn default)

const afterCorrect = updateBKT(initial, true);
console.log(afterCorrect.pKnow); // Should be > initial.pKnow (mastery increased)

const afterWrong = updateBKT(afterCorrect, false);
console.log(afterWrong.pKnow); // Should be < afterCorrect.pKnow (mastery decreased)
```

**Expected BKT parameters (defaults):**
| Parameter | Default | Range |
|---|---|---|
| `pKnow` (prior) | 0.3 | 0–1 |
| `pLearn` | 0.3 | 0–1 |
| `pSlip` | 0.1 | 0–0.3 |
| `pGuess` | 0.2 | 0–0.4 |

## 4. SM-2 Spaced Repetition Verification

Verify `src/lib/sm2.ts`:

```typescript
import { calculateSM2, createSM2Card } from './src/lib/sm2';

const card = createSM2Card('user1', 'flashcard1');
console.log(card.easiness);   // Should be 2.5
console.log(card.interval);   // Should be 1 (day)

const afterGood = calculateSM2(card, 4); // Quality 4 = good recall
console.log(afterGood.interval);  // Should be > 1
console.log(afterGood.easiness);  // Should be ≥ 2.5 for quality ≥ 4

const afterBad = calculateSM2(card, 1); // Quality 1 = forgot
console.log(afterBad.interval);   // Should reset to 1
```

## 5. xpToLevel Utility Verification

Verify consistent level computation in `src/lib/utils.ts`:

```typescript
import { xpToLevel } from './src/lib/utils';

console.log(xpToLevel(0));     // { level: 1, current: 0, needed: 100 }
console.log(xpToLevel(100));   // { level: 2, ... }
console.log(xpToLevel(5000));  // Higher level
```

All XP → level conversions in Quiz, Flashcards, and Dashboard should use this single function to stay consistent.

## 6. Offline Behaviour Testing

1. Open DevTools → **Network** tab → set throttle to **Offline**
2. Navigate to a chapter you've previously opened (cached in IndexedDB)
3. ✅ Chapter notes should load from cache
4. ✅ Offline banner should appear at the top of the chapter view
5. ✅ Quiz should still load (questions cached or generated by offline engine)
6. ✅ TTS button should show an error toast (not a blocking `alert()`)
7. ✅ Completing a quiz should save results to IndexedDB successfully

---

## 7. ErrorBoundary Testing

To manually trigger the ErrorBoundary recovery UI:

1. Open browser console
2. Navigate to any page
3. Execute: `window.__triggerError = true` (or throw inside a component temporarily)
4. ✅ Error boundary should display "Something went wrong" with a Reload button
